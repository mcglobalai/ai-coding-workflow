# 2026-04-08 讨论：AI驱动开发流程框架

## 背景

公司团队决定全面采用AI驱动开发，希望制定标准化的流程与框架，实现人与人、人与AI、AI与AI之间的交互过程。

## 已确定的六阶段流程（D002）

```
0 → Spec(需求分析) → TDD → Spec(开发需求) → Code → Test
```

| 阶段 | 名称 | 性质 |
|------|------|------|
| 1 | 0 | 起点 |
| 2 | Spec (需求分析) | 输入，明确要做什么 |
| 3 | TDD | **协同核心**，团队对齐的锚点 |
| 4 | Spec (开发需求) | 中间状态，开发需求拆解 |
| 5 | Code | 编码实现 |
| 6 | Test | 测试验证 |

### 关键区分

- **TDD = 协同核心** — 团队需要对齐的关键节点
- **Spec = 中间状态** — 个人连续工作的产物，非协同关键点

### 阶段一：0 → Spec(需求分析)

流程：
1. 使用者通过 Claude Code 整理需求文档
2. Claude Code 交给 Codex 进行审核
3. 审核结果返回给 Claude Code
4. Claude 基于 GPT 的结果判断：take / 部分take / 拒绝（需附理由）
5. 若部分拒绝，将不同意部分返回 GPT 再看一遍
6. Claude 接受（若仍有问题，需人介入审核）

### 阶段二：Spec(需求分析) → TDD

流程：
1. 由专人在本地AI上从头到尾生成
2. 基于git实现协同，让团队成员对TDD达成一致
3. TDD达成一致后，开发方向基本达成一致

### 阶段三：TDD → Spec(开发需求)

从 TDD 拆解出开发需求，作为后续开发工作的输入。

### 阶段四：Spec(开发需求) → Code

由 PPM AI Agent 统筹管理开发任务。

### 阶段五：Code → Test

由 PPM AI Agent 统筹管理，形成闭环。

## PPM AI Agent 的角色扩展

- **原定位**：统筹 TDD→Code→Test 阶段
- **扩展意向**：加入前两阶段（0→Spec、Spec→TDD）统计人的执行状态和计划

## 现有实现情况

- 已开发 PPM agent plugin（不在 OpenClaw 的 agent 里）
- 使用情况：没太用起来
- 问题感知：前置步骤缺少、使用上有限制

## 新增材料：AI-Native Development Workflow

来源：`inbox/ai_native_dev_workflow.md`、`inbox/ckg_tdd_template_v2.md`

### 核心理念

> Human writes decisions. AI writes everything else.

### 流程定义

```
Human → TDD (decisions + contract + rules + examples)
           ↓
         AI → Spec → Code → Tests
           ↓
Human → Verify against TDD (not against code)
```

### TDD 文档结构（七段式）

| Section | Who writes | Purpose |
|---|---|---|
| Intent | Human | Why this task exists (2-3 sentences) |
| Decisions | Human | Judgment calls AI must not re-decide |
| Contract | Human | Exact input/output types — the boundary |
| Rules | Human | Testable behaviors (each → one assert) |
| Examples | Human | Hand-verified calibration points (2-3) |
| Done When | Human | Acceptance checklist |
| Not This | Human | Scope boundaries to prevent AI expansion |

### 与之前讨论的差异

| 对比项 | 之前讨论 | 新材料 |
|--------|----------|--------|
| 阶段顺序 | 0→Spec→TDD→Code→Test | Human→TDD→Spec→Code→Test |
| TDD 含义 | Tech Design Document | 契约文档（Decisions+Contract+Rules+Examples） |
| TDD 位置 | Spec 之后 | Spec 之前（人写TDD，AI生成Spec） |
| 验证对象 | 未明确 | 对TDD验证，非对code |

### 可借鉴内容

1. **TDD 七段式结构**：特别是 Decisions（人决策）和 Not This（边界约束）
2. **AI Spec Generation Instructions**：告诉AI如何解读TDD各部分
3. **验证策略**：Verify against TDD, not against code
4. **分工原则**：Human attention → judgment calls; AI attention → expansion work

### 潜在冲突

1. **阶段顺序矛盾**：TDD 在 Spec 之前还是之后？
2. **TDD 定义矛盾**：Tech Design Document vs 契约文档
3. **决策来源**：之前说Spec由Claude+Codex协作完成，新材料说TDD完全由人编写

## Decisions

### D001: TDD 定义

**决策**：TDD = Tech Design Document（技术设计文档）

"契约文件"是旧叫法，之后全部统一采用 "Tech Design Document" 的叫法。

### D002: 六阶段流程

**决策**：流程为六阶段

```
0 → Spec(需求分析) → TDD → Spec(开发需求) → Code → Test
```

**关键洞察**：
- TDD 是协同核心
- Spec 是中间状态（非协同关键点）
